## **🏛️ Architecture**

This project strictly adheres to a **Clean Architecture** pattern to separate concerns, making the codebase highly testable and easy to maintain:

1. **Routes (routes/)**: Act as the entry points. They receive HTTP requests, rely on Pydantic for validation, and delegate processing.  
2. **Controllers (Controller/)**: Contain the core business logic. They orchestrate data flow and format the final API responses.  
3. **Services (services/)**: The data access layer. All database interactions (currently simulated in-memory) occur exclusively here.

## **✨ Key Features**

* **Strict Type Checking & Validation**: Leverages Pydantic V2 for automatic, out-of-the-box request validation (e.g., proper email formatting, character limits).  
* **Standardized API Responses**: Implements a universal { status, message, data } wrapper for highly predictable frontend integration.  
* **Global Exception Handling**: Gracefully catches 422 Unprocessable Entity errors and reformats them into readable, user-friendly messages.  
* **Auto-Generated Documentation**: Native support for Swagger UI (/docs) and ReDoc (/redoc).

## **💻 Installation & Setup**

1. **Clone the repository**  
   git clone \<your-repository-url\>  
   cd FastAPI\_user

2. **Set up a Virtual Environment**  
   python \-m venv .venv

   \# Activate on Windows:  
   .venv\\Scripts\\activate  
   \# Activate on macOS/Linux:  
   source .venv/bin/activate

3. **Install Dependencies**  
   pip install \-r requirements.txt

   *(Note: The pydantic\[email\] extra is required for EmailStr validation).*  
4. **Run the Application**  
   uvicorn main:app \--reload

   The server will start at http://127.0.0.1:8000.

## **📡 API Reference**

Once the server is running, navigate to [http://127.0.0.1:8000/docs](http://127.0.0.1:8000/docs) to interact with the Swagger UI.

### **Endpoints**

| HTTP Method | Endpoint | Description |
| :---- | :---- | :---- |
| GET | /users | Retrieve a list of all users. |
| GET | /users/{user\_id} | Retrieve a specific user by their ID. |
| POST | /users | Create a new user account. |
| PUT | /users/{user\_id} | Update an existing user's details. |
| DELETE | /users/{user\_id} | Delete a user from the system. |

## **🛡️ Data Validation**

When creating (POST) or updating (PUT) a user, the API enforces the following strict schema:

{  
  "id": 1,  
  "name": "Jane Doe",           // Required: min length of 2 characters  
  "email": "jane@example.com",  // Required: must be a valid email string  
  "number": 9876543210          // Required: must be a 10-digit integer  
}

If validation fails, the API intercepts the request and returns a standardized 400 Bad Request explaining exactly which field failed.

## **🔄 Standardized Responses**

To eliminate guesswork for frontend applications (React, Vue, etc.), **every** response generated by this API follows a strict wrapper format.

**Success Response Example:**

{  
  "status": "success",  
  "message": "User created successfully",  
  "data": {  
    "id": 1,  
    "name": "Jane Doe",  
    "email": "jane@example.com",  
    "number": 9876543210  
  }  
}

**Error Response Example:**

{  
  "status": "error",  
  "message": "User not found",  
  "data": null  
}
